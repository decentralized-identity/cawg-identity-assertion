=== Trusted assertion trust model

IMPORTANT: This section augments the _<<_trust_model,trust model>>_ described elsewhere in this specification. It describes only the trust relationships that are specific to *<<_trusted_assertion,trusted assertions>>.*

Trust in a *<<_trusted_assertion,trusted assertion>>* is established entirely through configuration held by the validator.
A validator maintains a *<<_trusted_assertion_authority_registry,trusted assertion authority registry>>* — a list of *<<_trusted_assertion_authority,trusted assertion authorities>>* whose statements it is willing to surface — together with the public keys used to verify each authority’s signatures.

A *<<_trusted_assertion,trusted assertion>>* carries no certificate chain and no trust anchor.
The chain of trust is therefore _not_ contained in the credential; it is supplied out of band by the registry.
This is a deliberate design choice that keeps assertions compact and keeps validation fully offline.

==== Trusted assertion authority registry

Each entry in a *<<_trusted_assertion_authority_registry,trusted assertion authority registry>>* describes a single *<<_trusted_assertion_authority,authority>>* and MUST contain at least the following information:

* an `Authority ID`, which is the value that appears in the `iss` claim of assertions issued by this *<<_trusted_assertion_authority,authority>>;*
* a human-readable `Name` for the *<<_trusted_assertion_authority,authority>>;* and
* one or more public keys, each OPTIONALLY associated with a key identifier (`kid`), used to verify the COSE signatures of assertions issued by this *<<_trusted_assertion_authority,authority>>.*

.Illustrative registry contents (non-normative)
[source]
----
Authority ID: iptc
Name: IPTC
Public keys:
  - kid: key1
  - kid: key2

Authority ID: reuters
Name: Reuters
Public keys:
  - kid: key1
----

Listing more than one public key per *<<_trusted_assertion_authority,authority>>* allows an *<<_trusted_assertion_authority,authority>>* to rotate keys without invalidating previously issued assertions.
When combined with a trusted time stamp (see xref:_verify_the_time_stamp_and_validity_window[xrefstyle=full]), this supports historical validation: an assertion signed with a key that was valid at the time of signing can continue to validate after that key has been retired, provided the key remains present in the registry.

==== Distribution of registry contents

The means by which a validator obtains and updates its *<<_trusted_assertion_authority_registry,trusted assertion authority registry>>* is out of scope for this section, but the following requirements apply:

* Registry distribution MUST be performed out of band, independently of any individual *<<_trusted_assertion,trusted assertion>>.*
* A validator MUST NOT infer trust in an *<<_trusted_assertion_authority,authority>>* from the content of an assertion alone.
* Validation of a *<<_trusted_assertion,trusted assertion>>* MUST NOT require contacting the issuing *<<_trusted_assertion_authority,authority>>* or any other network service.

CAWG governs the format used to distribute *<<_trusted_assertion_authority,authority>>* identifiers and public keys, as described in xref:_trusted_assertion_governance[xrefstyle=full].

[NOTE]
.Decentralized authority identifiers
====
_This section is non-normative._

This draft treats the `Authority ID` as an opaque identifier assigned within a CAWG-governed registry.
Whether these identifiers could instead be expressed in a decentralized form (for example, as xref:_w3c_decentralized_identifier[decentralized identifiers]) is an open question for a future revision.
====
