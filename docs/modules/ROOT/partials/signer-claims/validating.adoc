=== Validating a signer claim

A validator SHOULD validate each *<<_signer_claim,signer claim>>* only after it has validated the enclosing *<<_identity_assertion,identity assertion>>* as described in xref:_validation_method[xrefstyle=full].
Validation of a *<<_signer_claim,signer claim>>* MUST be performed entirely offline: it MUST NOT require contacting the issuing *<<_signer_claim_authority,authority>>* or any other network service.

The validator SHALL follow the steps in the following subsections, in the order presented, for each *<<_signer_claim,signer claim>>* it processes.

==== Validate the CAWG credential and extract the subject identifier

The validator SHALL first complete validation of the _<<_named_actor,named actor’s>>_ _<<_verifiable_identifier,verifiable identifier>>_ as described elsewhere in this specification.
From that credential, the validator SHALL extract the subject identifier of the _<<_named_actor,named actor>>_ (for example, the subject distinguished name of an X.509 certificate).
This identifier is used in xref:_verify_the_subject_binding[xrefstyle=full].

==== Parse the `COSE_Sign1` structure

The validator SHALL parse the *<<_signer_claim,signer claim>>* as a tagged `COSE_Sign1` object as described in link:++https://www.rfc-editor.org/rfc/rfc9052.html#name-signing-with-one-signer++[Section 4.2, “Signing with one signer,”] of RFC 9052.
If parsing fails, the validator MUST stop validation of this claim and issue the failure code `cawg.signer_claim.invalid_cose_sign1`.

The validator SHALL inspect the protected header `alg` parameter.
If the `alg` parameter is missing or references an algorithm that is not one of those defined in link:++https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html#_digital_signatures++[Section 13.2, “Digital signatures,”] of the C2PA technical specification, the validator MUST issue the failure code `cawg.signer_claim.invalid_alg` but MAY continue validation.

==== Parse the CWT claims set

The validator SHALL obtain the `COSE_Sign1` payload and parse it as a CWT claims set as described in link:https://datatracker.ietf.org/doc/html/rfc8392#section-3[Section 3, “Claims,”] of RFC 8392.
If the payload cannot be parsed, or if any of the required claims (`iss`, `sub`, `nbf`, `exp`, or the assertion claim type) is absent, the validator MUST stop validation of this claim and issue the failure code `cawg.signer_claim.invalid_cwt`.

==== Identify the issuing authority

The validator SHALL inspect the `iss` claim and locate a matching `Authority ID` in its *<<_signer_claim_authority_registry,signer claim authority registry>>.*
If no matching *<<_signer_claim_authority,authority>>* is found, the validator MUST stop validation of this claim and issue the failure code `cawg.signer_claim.untrusted_authority`.

NOTE: A *<<_signer_claim,signer claim>>* issued by an *<<_signer_claim_authority,authority>>* that is not in the registry is not an error in the claim itself. A validator SHOULD treat `cawg.signer_claim.untrusted_authority` as a signal that the statement should not be surfaced to the user, rather than as a defect in the enclosing *<<_identity_assertion,identity assertion>>.*

==== Select the authority public key

The validator SHALL select the public key or keys used to verify the signature from the registry entry located in the previous step.
If the `COSE_Sign1` header contains a `kid` parameter, the validator SHALL select the key whose identifier matches that `kid`.
If the `kid` does not match any key listed for the *<<_signer_claim_authority,authority>>,* the validator MUST issue the failure code `cawg.signer_claim.unknown_key` but MAY continue validation using the other keys listed for the *<<_signer_claim_authority,authority>>.*
If no `kid` is present, the validator SHALL attempt verification against each key listed for the *<<_signer_claim_authority,authority>>.*

==== Verify the COSE signature

The validator SHALL verify the signature using the selected public key material as described in link:++https://www.rfc-editor.org/rfc/rfc9052#section-4.4++[Section 4.4, “Signing and verification process,”] of RFC 9052.
If the signature does not verify against any candidate key, the validator MUST issue the failure code `cawg.signer_claim.signature_mismatch` and MUST NOT surface the statement carried by this claim.

==== Verify the subject binding

The validator SHALL compare the `sub` claim of the CWT against the subject identifier extracted from the _<<_named_actor,named actor’s>>_ _<<_verifiable_identifier,verifiable identifier>>_ in xref:_validate_the_cawg_credential_and_extract_the_subject_identifier[xrefstyle=full].
If the two do not correspond, the validator MUST issue the failure code `cawg.signer_claim.subject_mismatch` and MUST NOT surface the statement carried by this claim.

This step guarantees that a *<<_signer_claim,signer claim>>* can only be applied to the _<<_named_actor,named actor>>_ it was issued about, and cannot be transplanted onto a different signer.

==== Verify the time stamp and validity window

The validator SHALL determine a trusted reference time for this claim.
The reference time SHALL be taken, in order of preference, from:

* the time stamp carried in the *<<_signer_claim,signer claim’s>>* own `sigTst2` unprotected header, if present and valid;
* the time stamp for the _<<C2PA Manifest>>_ as described in link:++https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html#_time_stamps++[Section 11.3.2.5, “Time-stamps,”] of the C2PA technical specification; or
* the current date and time, if no trusted time stamp is available.

If a `sigTst2` time stamp is present, the validator SHALL validate it following the procedure described in link:++https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html#_time_stamps++[Section 11.3.2.5, “Time-stamps,”] of the C2PA technical specification.
If the time stamp validates, the validator MUST issue the success code `cawg.signer_claim.time_stamp.validated`.
If it does not validate, the validator MUST issue the status code `cawg.signer_claim.time_stamp.invalid`, MUST NOT use the time stamp as the reference time, and SHALL fall back to the next available source.

Using the trusted reference time, the validator SHALL verify that:

[source]
----
nbf <= reference_time < exp
----

If `reference_time` is earlier than the `nbf` claim, the validator MUST issue the failure code `cawg.signer_claim.not_yet_valid` and MUST NOT surface the statement carried by this claim.
If `reference_time` is at or after the `exp` claim, the validator MUST issue the failure code `cawg.signer_claim.expired` and MUST NOT surface the statement carried by this claim.

Because the reference time is derived from a trusted time stamp when one is available, a claim that was valid at the time the _<<C2PA asset>>_ was signed continues to validate afterwards, even after its `exp` time has passed relative to the current date.

==== Surface the statement

If the validator has completed all of the preceding steps for a *<<_signer_claim,signer claim>>* without issuing any failure code, it MUST issue the success code `cawg.signer_claim.valid` for that claim.

Only at this point MAY the validator surface the statement carried by the claim to the _<<_identity_assertion_consumer,identity assertion consumer>>._
The validator MAY use the optional metadata described in xref:_signer_claim_metadata[xrefstyle=full] to present a friendly label, description, or icon for the *<<_assertion_claim_type,assertion claim type>>.*

==== Status codes

|=======================
| Value | Status | Definition

| `cawg.signer_claim.valid`

| The *<<_signer_claim,signer claim>>* passed all validation requirements and its statement may be surfaced.

| `cawg.signer_claim.time_stamp.validated`
| Success
| The time stamp carried by the *<<_signer_claim,signer claim>>* was validated.

| Failure

| `cawg.signer_claim.untrusted_authority`
| Failure
| The `iss` claim does not match any `Authority ID` in the validator’s *<<_signer_claim_authority_registry,signer claim authority registry>>.*

| `cawg.signer_claim.unknown_key`
| Failure
| The `kid` header does not match any public key listed for the issuing *<<_signer_claim_authority,authority>>.*

| `cawg.signer_claim.signature_mismatch`
| Failure

| `cawg.signer_claim.expired`

| `cawg.signer_claim.time_stamp.invalid`
| Failure
