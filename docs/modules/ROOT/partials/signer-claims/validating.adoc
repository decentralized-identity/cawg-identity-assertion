=== Validating a Named Actor claim

A validator MUST validate each *<<_named_actor_claim,Named Actor claim>>*, only after it has validated the enclosing *<<_identity_assertion,identity assertion>>* as described in xref:_validation_method[xrefstyle=full]. Validation of a *<<_named_actor_claim,Named Actor claim>>* MUST be performed entirely offline: it MUST NOT require contacting the issuing *<<_named_actor_claim_authority,authority>>* or any other network service.

The validator SHALL follow the steps in the following subsections, in the order presented, for each *<<_named_actor_claim,Named Actor claim>>* it processes.

==== Validate the CAWG credential and extract the subject identifier

The validator SHALL first complete validation of the Identity Assertion as described in xref:_validating_the_identity_assertion[xrefstyle=full].

==== Parse the `COSE_Sign1` structure in the Named Actor claim

The validator SHALL parse the *<<_named_actor_claim,Named Actor claim>>* as a tagged `COSE_Sign1` object as described in link:++https://www.rfc-editor.org/rfc/rfc9052.html#name-signing-with-one-signer++[Section 4.2, “Signing with one signer,”] of RFC 9052. If parsing fails, the validator MUST stop validation of this claim and issue the failure code `cawg.named_actor_claim.invalid_cose_sign1`.

The validator SHALL inspect the protected header `alg` parameter. If the `alg` parameter is missing or references an algorithm that is not one of those defined in link:++https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html#_digital_signatures++[Section 13.2, “Digital signatures,”] of the C2PA specification, the validator MUST issue the failure code `cawg.named_actor_claim.invalid_alg`.

==== Parse the Verifiable Credential payload

The validator SHALL obtain the `COSE_Sign1` payload and parse it as a CBOR-encoded verifiable credential. If the payload cannot be parsed, or if any of the required fields (`issuer`, `credentialSubject.id`, `credentialSubject.claims`, `validFrom`, or `validUntil`) is absent, the validator MUST stop validation of this claim and issue the failure code `cawg.named_actor_claim.invalid_vc`.

The validator SHALL verify that `credentialSubject` and `cawg:credentialBinding` are non-empty objects. If these requirements are not met, the validator MUST stop validation of this claim and issue the failure code `cawg.named_actor_claim.invalid_triples`.

==== Identify the issuing authority

The validator SHALL inspect the `issuer` field and locate a matching `Authority ID` in its *<<_named_actor_claim_authority_registry,Named Actor claim authority registry>>.*
If no matching *<<_named_actor_claim_authority,authority>>* is found, the validator MUST stop validation of this claim and issue the failure code `cawg.named_actor_claim.untrusted_authority`.

NOTE: A *<<_named_actor_claim,Named Actor claim>>* issued by an *<<_named_actor_claim_authority,authority>>* that is not in the registry is not an error in the claim itself. A validator SHOULD treat `cawg.named_actor_claim.untrusted_authority` as a signal that the statement should not be surfaced to the user, rather than as a defect in the enclosing *<<_identity_assertion,identity assertion>>.*

==== Select the authority public key

The validator SHALL select the public key or keys used to verify the signature from the registry entry located in the previous step.
If the `COSE_Sign1` header contains a `kid` parameter, the validator SHALL select the key whose identifier matches that `kid`.
If the `kid` does not match any key listed for the *<<_named_actor_claim_authority,authority>>,* the validator MUST issue the failure code `cawg.named_actor_claim.unknown_key` but MAY continue validation using the other keys listed for the *<<_named_actor_claim_authority,authority>>.*
If no `kid` is present, the validator SHALL attempt verification against each key listed for the *<<_named_actor_claim_authority,authority>>.*

==== Verify the COSE signature

The validator SHALL verify the signature using the selected public key material as described in link:++https://www.rfc-editor.org/rfc/rfc9052#section-4.4++[Section 4.4, “Signing and verification process,”] of RFC 9052.
If the signature does not verify against any candidate key, the validator MUST issue the failure code `cawg.named_actor_claim.signature_mismatch` and MUST NOT surface the statement carried by this claim.

==== Verify the subject binding

The validator SHALL compare `credentialSubject.id` against the subject identifier extracted from the _<<_named_actor,named actor’s>>_ _<<_verifiable_identifier,verifiable identifier>>_ in xref:_validate_the_cawg_credential_and_extract_the_subject_identifier[xrefstyle=full].
If the two do not correspond, the validator MUST issue the failure code `cawg.named_actor_claim.subject_mismatch` and MUST NOT surface the statement carried by this claim.

This step guarantees that a *<<_named_actor_claim,Named Actor claim>>* can only be applied to the _<<_named_actor,named actor>>_ it was issued about, and cannot be transplanted onto a different signer.

[[_verify_credential_binding]]
==== Verify credential binding

The validator MUST verify that the `cawg:credentialBinding` field binding matches the signing key of the enclosing *<<_identity_assertion,identity assertion>>.*

For a binding of type `cawg:x509SpkiSha256`, the validator SHALL:

. Extract the end-entity X.509 certificate from the *<<_identity_assertion,identity assertion>>* signature.
. Compute the SHA-256 digest of the DER-encoded SubjectPublicKeyInfo (SPKI) of that certificate and encode it as a base64url string (without padding).
. Compare the computed value against the `cawg:x509SpkiSha256` value in the credential.

If the values do not match, the validator MUST issue the failure code `cawg.named_actor_claim.credential_binding_mismatch` and MUST NOT surface the statement carried by this claim.

If the `cawg:credentialBinding` map contains a binding type that the validator does not recognise, the validator MUST issue the failure code `cawg.named_actor_claim.credential_binding_unknown_type` and MUST NOT surface the statement carried by this claim.

==== Verify the time stamp and validity window

The validator SHALL determine a trusted reference time for this claim.
The reference time SHALL be taken, in order of preference, from:

* the time stamp carried in the *<<_named_actor_claim,Named Actor claim’s>>* own `sigTst2` unprotected header, if present and valid;
* the time stamp for the _<<C2PA Manifest>>_ as described in link:++https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html#_time_stamps++[Section 11.3.2.5, “Time-stamps,”] of the C2PA technical specification; or
* the current date and time, if no trusted time stamp is available.

If a `sigTst2` time stamp is present, the validator SHALL validate it following the procedure described in link:++https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html#_time_stamps++[Section 11.3.2.5, “Time-stamps,”] of the C2PA technical specification.
If the time stamp validates, the validator MUST issue the success code `cawg.named_actor_claim.time_stamp.validated`.
If it does not validate, the validator MUST issue the status code `cawg.named_actor_claim.time_stamp.invalid`, MUST NOT use the time stamp as the reference time, and SHALL fall back to the next available source.

Using the trusted reference time, the validator SHALL verify that:

[source]
----
validFrom <= reference_time < validUntil
----

If `reference_time` is earlier than `validFrom`, the validator MUST issue the failure code `cawg.named_actor_claim.not_yet_valid` and MUST NOT surface the statement carried by this claim.
If `reference_time` is at or after `validUntil`, the validator MUST issue the failure code `cawg.named_actor_claim.expired` and MUST NOT surface the statement carried by this claim.

Because the reference time is derived from a trusted time stamp when one is available, a claim that was valid at the time the _<<C2PA asset>>_ was signed continues to validate afterwards, even after its `validUntil` time has passed relative to the current date.

[[_verify_credential_status]]
==== Verify credential status

If a Named Actor claim contains a `credentialStatus` field, the validator MUST evaluate it using locally available status data.
Validation of credential status MUST NOT require network access at claim-validation time.

If status data is unavailable, stale, or insufficient to determine the status of the claim, the validator MUST issue the failure code `cawg.named_actor_claim.status_unknown` and MUST NOT surface the statement carried by this claim.

If status data indicates that the claim is revoked or suspended, the validator MUST issue the failure code `cawg.named_actor_claim.revoked` and MUST NOT surface the statement carried by this claim.

==== Surface the statement

If the validator has completed all of the preceding steps for a *<<_named_actor_claim,Named Actor claim>>* without issuing any failure code, it MUST issue the success code `cawg.named_actor_claim.valid` for that claim.

Only at this point the validator output must include the statement carried by the claim to the _<<_identity_assertion_consumer,identity assertion consumer>>._
The validator MAY use the optional metadata described in xref:_named_actor_claim_metadata[xrefstyle=full] to present a friendly label, description, or icon for the *<<_assertion_claim_type,assertion claim type>>.*
