=== Signer claim credential format

A *<<_signer_claim,signer claim>>* is a link:https://datatracker.ietf.org/doc/html/rfc8392[CBOR Web Token (CWT), RFC 8392] secured as a `COSE_Sign1` structure as described in link:++https://www.rfc-editor.org/rfc/rfc9052.html#name-signing-with-one-signer++[Section 4.2, “Signing with one signer,”] of RFC 9052.

Each *<<_signer_claim,signer claim>>* is a self-contained, signed CWT.
It MUST NOT embed a certificate chain or any trust anchor material.
The public key required to verify the signature is obtained by the validator from its *<<_signer_claim_authority_registry,signer claim authority registry>>* as described in xref:_signer_claim_trust_model[xrefstyle=full], not from the claim itself.

[NOTE]
.Expected size
====
_This section is non-normative._

Because a *<<_signer_claim,signer claim>>* carries no certificate chain and no trust anchors, a typical claim is expected to occupy roughly 100–200 bytes.
This keeps the impact on overall _<<C2PA Manifest>>_ size small even when several claims are present.
====

==== COSE profile

The `COSE_Sign1` protected header of a *<<_signer_claim,signer claim>>* MUST include an `alg` (label 1) parameter whose value is one of the signature algorithms defined in link:++https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html#_digital_signatures++[Section 13.2, “Digital signatures,”] in version 2.4 or any subsequent version of the C2PA technical specification.

The `COSE_Sign1` header MAY include a `kid` (key identifier, label 4) parameter.
If present, the `kid` identifies which of the issuing *<<_signer_claim_authority,authority’s>>* public keys was used to produce the signature, so that a validator can select the correct key during a key-rollover period.
If the `kid` is absent, the validator SHALL attempt verification against each public key that the *<<_signer_claim_authority_registry,registry>>* lists for the issuing *<<_signer_claim_authority,authority>>.*

A *<<_signer_claim,signer claim>>* MAY carry a link:https://datatracker.ietf.org/doc/html/rfc3161[RFC 3161]-compliant time stamp in a `sigTst2` unprotected header, generated as described by link:++https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html#_time_stamps++[Section 11.3.2.5, “Time-stamps,”] of the C2PA technical specification.
If a time stamp is present, it MUST be a version 2 or later time stamp; a version 1 time stamp MUST NOT be used.

==== CWT claims

The `COSE_Sign1` payload MUST be a CWT claims set as defined in link:https://datatracker.ietf.org/doc/html/rfc8392#section-3[Section 3, “Claims,”] of RFC 8392.
The following claims are defined for a *<<_signer_claim,signer claim>>:*

[width="100%",cols="4,3,3,10",options="header"]
|=======================
| Claim | Key | Presence | Meaning

| Issuer (`iss`)
| 1
| Required
| The identifier of the *<<_signer_claim_authority,signer claim authority>>* that issued this claim. This value MUST match an `Authority ID` in the validator’s *<<_signer_claim_authority_registry,signer claim authority registry>>* for the claim to be trusted.

| Subject (`sub`)
| 2
| Required
| The CAWG entity identifier of the _<<_named_actor,named actor>>_ that this claim is about. This value MUST correspond to the subject identifier of the _<<_named_actor,named actor’s>>_ _<<_verifiable_identifier,verifiable identifier>>_ as described in xref:_verify_the_subject_binding[xrefstyle=full].

| Not before (`nbf`)
| 5
| Required
| The time before which the claim MUST NOT be considered valid, expressed as described in link:https://datatracker.ietf.org/doc/html/rfc8392#section-3[RFC 8392].

| Expiry (`exp`)
| 4
| Required
| The time at or after which the claim MUST NOT be considered valid.

| Issued at (`iat`)
| 6
| Optional
| The time at which the claim was issued.

| CWT ID (`cti`)
| 7
| Optional
| A unique identifier for this claim, which MAY be used by an *<<_signer_claim_authority,authority>>* for its own record keeping.

| Assertion claim type
| `cawg.typ`
| Required
| A globally unique *<<_assertion_claim_type,assertion claim type>>* identifier that names the statement being made. See xref:_assertion_claim_type_syntax[xrefstyle=full].

|=======================

[IMPORTANT]
.Provisional claim key
====
The concrete registration of the assertion claim type key is expected to be finalized before this section leaves draft status.
This draft uses the text-string key `cawg.typ`, which is consistent with the label rules described in xref:_labels[xrefstyle=full].
A future revision MAY replace this with a CWT integer claim key registered with IANA.
====

===== Assertion claim type syntax

The assertion claim type identifier names the statement carried by a *<<_signer_claim,signer claim>>* (for example, `iptc:vnp` or `reuters:member`).

The identifier is a namespaced value controlled by the issuing *<<_signer_claim_authority,authority>>.*
CAWG standardizes only the _syntax_ of the identifier and the requirement that it be globally unique; it does not standardize the set of claim types.

An assertion claim type identifier MUST be a non-empty string consisting of an authority-controlled namespace, a `:` separator, and a namespace-specific value:

[source,abnf]
.ABNF for assertion claim type identifiers
----
claim-type   = namespace ":" specific
namespace    = 1*( DIGIT / ALPHA ) *( DIGIT / ALPHA / "-" / "_" )
specific     = 1*( DIGIT / ALPHA ) *( DIGIT / ALPHA / "-" / "_" / ":" )
----

The `namespace` component identifies the *<<_signer_claim_authority,authority>>* that governs the meaning of the identifier.
It is RECOMMENDED that the `namespace` component match the `Authority ID` (the `iss` claim) of the issuing *<<_signer_claim_authority,authority>>.*

.Examples of assertion claim type identifiers
[%collapsible]
====
* `iptc:vnp`
* `iptc:news-agency`
* `ukgov:licensed-media`
* `reuters:member`
====

A validator MUST NOT hardcode knowledge of any particular assertion claim type.
The semantics of an assertion claim type are defined by the issuing *<<_signer_claim_authority,authority>>* and MAY be discovered through the optional mechanism described in xref:_signer_claim_metadata[xrefstyle=full].

==== Carriage in the identity assertion

// TODO: The exact carriage mechanism is an open design point and MUST be
// finalized (including a corresponding update to the CBOR schema in the
// assertion-definition section) before this section leaves draft status.

One or more *<<_signer_claim,signer claims>>* MAY be carried within the *<<_identity_assertion,identity assertion>>* that describes the same _<<_named_actor,named actor>>._
This draft proposes that they be carried in an OPTIONAL `signer_assertions` field added to the `identity` map, defined as a CBOR array in which each entry is a byte string containing one tagged `COSE_Sign1` CWT:

[source]
----
? "signer_claims": [1* bstr], ; each entry is a tagged COSE_Sign1 CWT
----

The `signer_claims` field is independent of the `signer_payload` and `signature` fields.
A *<<_signer_claim,signer claim>>* is signed by the issuing *<<_signer_claim_authority,authority>>,* not by the _<<_credential_holder,credential holder>>,_ and its validity is established independently of the _<<_credential_holder,credential holder’s>>_ signature.

A validator that does not support this mechanism, or that encounters an unknown *<<_signer_claim_authority,authority>>,* MUST ignore the affected *<<_signer_claim,signer claims>>* and MUST NOT treat their presence as a validation failure of the enclosing *<<_identity_assertion,identity assertion>>.*

[NOTE]
.Relationship to the CBOR schema
====
Adopting the `signer_claims` field requires a corresponding addition to the `identity` rule in xref:_cbor_schema[xrefstyle=full].
That change is intentionally deferred until the carriage mechanism is confirmed, so that the normative schema is not modified while this section remains in draft status.
====
