=== Named Actor claim credential format

A *<<_named_actor_claim,Named Actor claim>>* is a COSE signature object whose unencoded payload is an unsecured _<<_w3c_verifiable_credential,W3C verifiable credential>>_, following link:++https://www.w3.org/TR/vc-jose-cose/#securing-with-cose++[Section 3.3, "With COSE,"] of the W3C standard _Securing Verifiable Credentials using JOSE and COSE_ and link:++https://www.rfc-editor.org/rfc/rfc9052.html#name-signing-with-one-signer++[Section 4.2, "Signing with one signer,"] of RFC 9052.

Each *<<_named_actor_claim,Named Actor claim>>* is a self-contained, signed verifiable credential. It MUST NOT embed a certificate chain or any trust anchor material. The public key required to verify the signature is obtained by the validator from its *<<_named_actor_claim_authority_registry,Named Actor claim authority registry>>* as described in xref:_named_actor_claim_trust_model[xrefstyle=full], not from the claim itself.

==== COSE profile

The `COSE_Sign1` protected header of a *<<_named_actor_claim,Named Actor claim>>* MUST include an `alg` (label 1) parameter whose value is one of the signature algorithms defined in link:https://spec.c2pa.org/specifications/specifications/2.4/specs/C2PA_Specification.html#_digital_signatures[Section 13.2, “Digital signatures,”] in version 2.4 or any subsequent version of the C2PA technical specification.

The `COSE_Sign1` header MAY include a `kid` (key identifier, label 4) parameter. If present, the `kid` identifies which of the issuing *<<_named_actor_claim_authority,authority’s>>* public keys was used to produce the signature, so that a validator can select the correct key from the *<<_named_actor_claim_authority_registry,authority registry>>* instead of trying each key. If the `kid` is absent, the validator SHALL attempt verification against each public key that the *<<_named_actor_claim_authority_registry,authority registry>>* lists for the issuing *<<_named_actor_claim_authority,authority>>.*

==== Verifiable Credential claims

The `COSE_Sign1` payload MUST be the unsecured verifiable credential payload, encoded as CBOR-LD as defined by link:https://www.w3.org/TR/cbor-ld/[_CBOR-LD 1.0: A CBOR-based Serialization for Linked Data_]. The following fields are defined for a *<<_named_actor_claim,Named Actor claim>>:*

[width="100%",cols="4,3,10",options="header"]
|=======================
| Field | Presence | Meaning

| `issuer`
| Required
| The identifier of the *<<_named_actor_claim_authority,Named Actor claim authority>>* that issued this claim. This value MUST match the `Authority ID` of the credential and key used to sign it. See _<<_identify_the_issuing_authority,Identify the issuing authority>>_ for more detail.

| `validFrom`
| Required
| The time before which the claim MUST NOT be considered valid.

| `validUntil`
| Required
| The time at or after which the claim MUST NOT be considered valid.

| `id`
| Optional
| A unique identifier for this claim, which MAY be used by an *<<_named_actor_claim_authority,authority>>* for its own record keeping.

| `credentialSubject`
| Required
| This MUST contain one or more triples that carry the statement(s) made by this Named Actor claim.

| `credentialSubject.id`
| Optional
| The identifier of the _<<_named_actor,named actor>>_ that this claim is about. This value SHOULD be a URI. This value MAY correspond to the subject identifier of the _<<_named_actor,named actor’s>>_ _<<_verifiable_identifier,verifiable identifier>>_ but is not involved in <<_verify_the_subject_binding,verifying the subject binding>>. If the binding to the subject is verified and a value is present, a `sameAs` relationship MUST be asserted between the `credentialSubject.id` and the _<<_named_actor,named actor’s>>_ _<<_verifiable_identifier,verifiable identifier>>_. If the binding to the subject is verified and no value is present, the _<<_named_actor,named actor’s>>_ _<<_verifiable_identifier,verifiable identifier>>_ MUST be asserted as the `credentialSubject.id`.

| `credentialStatus`
| Required
| Status information that can be used to determine whether the claim has been revoked or suspended. If present, validators MUST evaluate it as described in xref:_verify_credential_status[xrefstyle=full].

| `cawg:credentialBinding`
| Required
| A map that cryptographically binds this credential to a specific signing key. A validator MUST verify this binding as described in xref:_verify_credential_binding[xrefstyle=full].

|=======================

[[_credential_binding_field]]
===== `cawg:credentialBinding` field

The `cawg:credentialBinding` field cryptographically binds the *<<_named_actor_claim,Named Actor claim>>* to the signing key of the enclosing *<<_identity_assertion,identity assertion>>*.

The field is a map whose key identifies the binding type and whose value is the binding material. The following example uses the only currently defined binding type:

[source,json]
----
"cawg:credentialBinding": { "cawg:x509SpkiSha256": "xyz123…" }
----

`cawg:x509SpkiSha256`::: The base64url-encoded SHA-256 digest (without padding) of the DER-encoded SubjectPublicKeyInfo (SPKI) of the end-entity X.509 certificate used to produce the *<<_identity_assertion,identity assertion>>* signature in which this *<<_named_actor_claim,Named Actor claim>>* is carried.

===== Authority-defined statements in `credentialSubject`

The statement carried by a *<<_named_actor_claim,Named Actor claim>>* MUST be represented as one or more authority-defined statements in the `credentialSubject` object.

The `credentialSubject` map MUST include at least one triple (that is, a set of JSON-LD fields and values that have a mapping in the JSON-LD context) which asserts a statement about the _<<_named_actor,named actor>>_ that is the subject of the claim. Each triple consists of three components:

* The triple `subject` is always `credentialSubject.id`. 
* The `predicate` value defines the specific property, trait, or relationship being asserted.
* The `object` value is the value asserted by the issuing *<<_named_actor_claim_authority,authority>>* for that predicate.

The `predicate` identifier MUST be either a namespaced value controlled by the issuing *<<_named_actor_claim_authority,authority>>*, or part of a standardised vocabulary. In either case, the vocabulary/namespace MUST be defined in the JSON-LD context of the Verifiable Credential.

Where the predicate or object value is namespaced, it is RECOMMENDED that the namespace match the `Authority ID` (the `issuer` value) of the issuing *<<_named_actor_claim_authority,authority>>*.

.Examples of triple predicate identifiers
[%collapsible]
====
* `type` (mapped in the Verifiable Credentials context to `@type`)
* `gov-broadcast:licensee`
* `journalist-association:member`
====

The semantics of each predicate and object are defined by the namespace and MAY be discovered through the optional mechanism described in xref:_named_actor_claim_metadata[xrefstyle=full].

==== Carriage in the identity assertion

One or more *<<_named_actor_claim,Named Actor claims>>* MAY be carried within the *<<_identity_assertion,identity assertion>>* that describes the same _<<_named_actor,named actor>>_. Such claims are carried in the optional `named_actor_claims` field added to the `identity` map, defined as a CBOR array in which each entry is a byte string containing one tagged `COSE_Sign1` Verifiable Credential:

[source]
----
? "named_actor_claims": [1* bstr], ; each entry is a tagged COSE_Sign1 VC
----

The `named_actor_claims` field is independent of the `signer_payload` and `signature` fields. A *<<_named_actor_claim,Named Actor claim>>* is signed by the issuing *<<_named_actor_claim_authority,authority>>,* not by the _<<_credential_holder,credential holder>>,_ and its validity is established independently of the _<<_credential_holder,credential holder’s>>_ signature (see xref:_validating_a_named_actor_claim[xrefstyle=full]).

A validator that does not support this mechanism, or that encounters an unknown *<<_named_actor_claim_authority,authority>>,* MUST ignore the affected *<<_named_actor_claim,Named Actor claims>>* and MUST NOT treat their presence as a validation failure of the enclosing *<<_identity_assertion,identity assertion>>*.
